# 13 — Database Migrations

Rule: **every schema change must be reversible and deployable while old code is still
running.** Schema mistakes are the hardest to undo — data, unlike code, has no reflog.

## Writing a migration

1. One migration per logical change; generated by the tool (alembic/prisma/knex/
   rails), then READ and edited — autogenerate lies (misses renames, sees drift).
2. Write the downgrade path. Can't be downgraded (data-destroying)? Say so explicitly
   in the migration and in your report — that's a decision, not a default.
3. **Guard against drift:** if the app ever auto-creates tables (`create_all`, dev
   sync modes), local schema runs ahead of migration history. Write inspector-guarded
   DDL so re-running is safe:
   ```python
   if "new_col" not in [c["name"] for c in inspect(conn).get_columns("t")]:
       op.add_column("t", sa.Column("new_col", sa.String(), nullable=True))
   ```
4. Test the round trip on a COPY of realistic data: upgrade → downgrade → upgrade.
5. Dialect check: migration must run on BOTH dev and prod engines (SQLite quirks:
   limited ALTER; Postgres: long locks on big tables).

## Expand → migrate → contract (the zero-downtime pattern)

Renames, type changes, moves, drops — never in one step. Old code keeps running
against the schema during deploy, so:

1. **Expand:** add the new column/table (nullable or defaulted). Deploy. Old code
   ignores it, nothing breaks.
2. **Migrate:** backfill data (batched: `UPDATE … WHERE id IN (next N)` in a loop —
   one giant UPDATE locks the table); new code writes both / reads new-then-old.
   Deploy.
3. **Contract:** once nothing reads the old column (verify: grep + logs), drop it in
   a LATER release.

Corollary: **never drop/rename a column in the same release as the code change that
stops using it.**

## New columns on existing tables

- Nullable, or with a server-side default. NOT NULL without default on a populated
  table = failed migration.
- Adding an index to a big table: use the engine's non-blocking mode
  (`CREATE INDEX CONCURRENTLY` in Postgres) — plain CREATE INDEX locks writes.

## Data backfills

- Idempotent (safe to re-run after a crash mid-way).
- Batched with progress logging.
- As a script kept in the repo — never ad-hoc SQL typed into a prod console. If a
  prod console is unavoidable: run inside a transaction, `SELECT` the affected rows
  first, count them, then write — and save the exact SQL in the PR/report.

## Bans

- Editing a migration that has ever run anywhere but your machine (write a new one).
- Destructive DDL (DROP/TRUNCATE) without a named backup or explicit human sign-off.
- Mixing schema change + big data backfill + code change in one deploy.
- Trusting `autogenerate` output unread.

## Output contract

Migration file reviewed line-by-line · downgrade written or explicitly declared
impossible · round-trip tested (output shown) · zero-downtime pattern used for any
rename/drop · backfill script idempotent and committed.
